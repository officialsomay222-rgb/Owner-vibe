## 2024-05-13 - Enhance A11y and UX for Settings UI Form Components
**Learning:** React fragments or loose `<span>` elements around form controls severely harm screen reader accessibility. By explicitly adding unique generated IDs via `React.useId()` and tying them to `<button>` through `aria-labelledby` and `aria-describedby` (as well as adding standard `<label htmlFor="...">`), standard form fields become significantly easier to use. Similarly, mobile focus outlines often require high-contrast color shifts (`focus-visible:ring-offset-2`) since background darkness levels can vary randomly depending on user/system app themes.
**Action:** When creating modular generic input wrappers (Toggle/Select/Action), inject automatic ID generation and explicitly pair titles, descriptions, and controls. Always test focus rings with dual offset properties to ensure visibility across dynamic backgrounds.
## 2024-05-19 - Improved MiniPlayer Interaction Accessibility
**Learning:** Attaching click handlers directly to visual wrapper elements (like `div`) creates significant accessibility barriers, as they lack semantic roles, do not receive keyboard focus, and aren't announced as interactive by screen readers. Furthermore, nesting interactive elements (like play/pause buttons) inside an interactive container can lead to complex event propagation issues.
**Action:** Replace interactive `div` containers with standard structural `div`s that contain a visually hidden, absolutely positioned `<button>` layer (`z-0 inset-0`) to handle the primary action. Use the CSS `:has()` pseudo-class on the wrapper (e.g., `has-[.inner-btn:active]:scale-[0.98]`) to maintain active state visual feedback. Ensure all nested inner buttons have a higher z-index (`z-10`) and properly defined `aria-label` and `title` attributes, along with `focus-visible` styles for keyboard navigation.
