## 2024-05-13 - Enhance A11y and UX for Settings UI Form Components
**Learning:** React fragments or loose `<span>` elements around form controls severely harm screen reader accessibility. By explicitly adding unique generated IDs via `React.useId()` and tying them to `<button>` through `aria-labelledby` and `aria-describedby` (as well as adding standard `<label htmlFor="...">`), standard form fields become significantly easier to use. Similarly, mobile focus outlines often require high-contrast color shifts (`focus-visible:ring-offset-2`) since background darkness levels can vary randomly depending on user/system app themes.
**Action:** When creating modular generic input wrappers (Toggle/Select/Action), inject automatic ID generation and explicitly pair titles, descriptions, and controls. Always test focus rings with dual offset properties to ensure visibility across dynamic backgrounds.

## 2025-05-18 - Nested Interactive Elements inside a Div Container
**Learning:** Placing nested `<button>`s inside an interactive `div` (with an `onClick`) creates accessibility and interaction issues because the parent wrapper itself isn't fully keyboard accessible and clicking inner buttons fires unexpected events.
**Action:** Instead of making the parent `div` interactive, wrap the parent's primary action in a visually hidden, absolutely positioned `<button>` placed behind the other elements. Use the CSS `:has()` selector (e.g., `has-[.hidden-btn:active]:scale-[0.98]`) on the container `div` to maintain visual click feedback when the user interacts with the hidden button layer. Make sure internal controls have a higher z-index and appropriate ARIA labels.

## 2025-05-18 - Nested Interactive Elements inside a Div Container - Part 2
**Learning:** Using `z-10` on inner containers inside an interactive `div` can create "dead zones" where clicking the padding/gap of the inner container prevents clicks from reaching the underlying elements (like the hidden button). Additionally, applying `opacity-0` to the hidden expand button prevents the browser's default focus ring from showing, violating keyboard accessibility guidelines.
**Action:** Always apply `pointer-events-none` to the inner container wrapper and `pointer-events-auto` strictly on the actual interactive buttons inside it. To ensure focus states are visible for screen reader/keyboard users, use CSS `:has()` with `:focus-visible` on the parent container (e.g., `has-[.hidden-btn:focus-visible]:ring-2`) to simulate the focus ring visually.
