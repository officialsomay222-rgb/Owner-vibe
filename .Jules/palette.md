## 2024-05-13 - Enhance A11y and UX for Settings UI Form Components
**Learning:** React fragments or loose `<span>` elements around form controls severely harm screen reader accessibility. By explicitly adding unique generated IDs via `React.useId()` and tying them to `<button>` through `aria-labelledby` and `aria-describedby` (as well as adding standard `<label htmlFor="...">`), standard form fields become significantly easier to use. Similarly, mobile focus outlines often require high-contrast color shifts (`focus-visible:ring-offset-2`) since background darkness levels can vary randomly depending on user/system app themes.
**Action:** When creating modular generic input wrappers (Toggle/Select/Action), inject automatic ID generation and explicitly pair titles, descriptions, and controls. Always test focus rings with dual offset properties to ensure visibility across dynamic backgrounds.

## 2026-05-17 - Accessible Complex Interactive Nested Components
**Learning:** Complex interactive components (like a clickable `div` that expands a player while also housing independent interactive buttons like Play/Pause) require explicit structural changes for accessibility. A standard HTML `div` lacks native keyboard interaction support. When applying an `onClick` to a container `div`, it must be accompanied by `role="button"`, `tabIndex={0}`, an `onKeyDown` handler (for 'Enter' and 'Space'), and an explicit `aria-label` to be usable by keyboard navigators and screen readers.
**Action:** Whenever converting a non-interactive element (like a `div` or `span`) into a primary clickable target with nested actions, always upgrade it to a full accessible interactive element by adding `role="button"`, `tabIndex={0}`, keyboard event handlers, and distinct `focus-visible` styling to prevent breaking keyboard navigation flows.

## 2026-05-17 - Avoid Invalid ARIA Nesting in Complex Components
**Learning:** Adding `role="button"` to a container `div` that already contains inner `<button>` elements creates invalid ARIA nesting. HTML and ARIA specifications strictly forbid nested interactive elements. Furthermore, simple `onKeyDown` handlers on the parent can inadvertently catch events bubbled from the inner controls if `e.stopPropagation()` is missing, breaking inner button functionality for keyboard users.
**Action:** Instead of making the parent container interactive, use an absolute positioned, visually hidden `<button>` layer (e.g. `absolute inset-0 w-full h-full z-0`) stretched across the container to handle the primary click action. Ensure other interactive inner elements are lifted above it (e.g. `z-10`) so they remain separate focusable targets without nested ARIA violations.

## 2026-05-17 - Modern CSS Selectors for Complex Focus/Active States
**Learning:** When separating nested interactive components (e.g. creating a visually hidden background button to fix ARIA nesting), the outer container often loses the `active` or `focus` visual feedback (like a container scale down on click) because those pseudo-classes now apply to the inner invisible button.
**Action:** Use CSS `has()` pseudo-class to apply styles to a parent container based on the state of its children. For example, `has-[.bg-expand-btn:active]:scale-[0.98]` allows the parent container to scale down precisely when the invisible inner background button is clicked, without triggering if a different inner element (like a play button) is clicked.
