## 2024-05-13 - Enhance A11y and UX for Settings UI Form Components
**Learning:** React fragments or loose `<span>` elements around form controls severely harm screen reader accessibility. By explicitly adding unique generated IDs via `React.useId()` and tying them to `<button>` through `aria-labelledby` and `aria-describedby` (as well as adding standard `<label htmlFor="...">`), standard form fields become significantly easier to use. Similarly, mobile focus outlines often require high-contrast color shifts (`focus-visible:ring-offset-2`) since background darkness levels can vary randomly depending on user/system app themes.
**Action:** When creating modular generic input wrappers (Toggle/Select/Action), inject automatic ID generation and explicitly pair titles, descriptions, and controls. Always test focus rings with dual offset properties to ensure visibility across dynamic backgrounds.
## 2024-05-24 - MiniPlayer: Refactoring Nested Interactive Elements
**Learning:** Attaching `onClick` handlers directly to container wrapper `div`s that also contain nested native `<button>` elements creates an accessibility anti-pattern where screen readers and keyboard navigation get confused by overlapping interactable targets. Furthermore, relying on `stopPropagation()` on the children to prevent parent triggers can be brittle.
**Action:** When a whole card or container needs to be clickable while also containing specific sub-actions (like playback controls), remove `onClick` from the parent. Instead, inject a visually hidden, absolutely positioned `<button>` (`inset-0 z-0 opacity-0`) as the primary container action. Use strict z-indexing and `pointer-events-none`/`auto` to layer the inner buttons over it. Finally, maintain hover/active styling on the visual parent by utilizing the CSS `:has()` pseudo-class (e.g., `has-[.expand-btn:active]:scale-[0.98]`).
