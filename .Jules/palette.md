## 2024-05-13 - Enhance A11y and UX for Settings UI Form Components
**Learning:** React fragments or loose `<span>` elements around form controls severely harm screen reader accessibility. By explicitly adding unique generated IDs via `React.useId()` and tying them to `<button>` through `aria-labelledby` and `aria-describedby` (as well as adding standard `<label htmlFor="...">`), standard form fields become significantly easier to use. Similarly, mobile focus outlines often require high-contrast color shifts (`focus-visible:ring-offset-2`) since background darkness levels can vary randomly depending on user/system app themes.
**Action:** When creating modular generic input wrappers (Toggle/Select/Action), inject automatic ID generation and explicitly pair titles, descriptions, and controls. Always test focus rings with dual offset properties to ensure visibility across dynamic backgrounds.

## 2024-05-19 - Fix nested interactive element a11y with hidden buttons and CSS :has()
**Learning:** Wrapping interactive nested elements inside a clickable parent `<div>` causes severe accessibility issues because screen readers and keyboard navigation (Tab) struggle to understand the semantic hierarchy. Additionally, placing an `onClick` on a `<div>` fails to provide native focus management and keyboard trigger support (Enter/Space).
**Action:** When a complex container needs to act as a button but also contains distinct nested interactive buttons (like a music mini-player), avoid adding `onClick` directly to the `<div>`. Instead, place a visually hidden `<button class="absolute inset-0 z-0">` that covers the container to capture primary clicks. Use the CSS `:has()` pseudo-class on the parent container (e.g., `has-[.expand-btn:active]:scale-[0.98]`) to bind visual active/focus feedback to the hidden button's state while ensuring nested controls (`z-10`) remain independently accessible and interactive.
