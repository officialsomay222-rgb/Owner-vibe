## 2024-05-13 - Enhance A11y and UX for Settings UI Form Components
**Learning:** React fragments or loose `<span>` elements around form controls severely harm screen reader accessibility. By explicitly adding unique generated IDs via `React.useId()` and tying them to `<button>` through `aria-labelledby` and `aria-describedby` (as well as adding standard `<label htmlFor="...">`), standard form fields become significantly easier to use. Similarly, mobile focus outlines often require high-contrast color shifts (`focus-visible:ring-offset-2`) since background darkness levels can vary randomly depending on user/system app themes.
**Action:** When creating modular generic input wrappers (Toggle/Select/Action), inject automatic ID generation and explicitly pair titles, descriptions, and controls. Always test focus rings with dual offset properties to ensure visibility across dynamic backgrounds.

## 2024-05-21 - Accessible Container Interaction with Nested Buttons
**Learning:** Making entire container `div`s interactive (e.g., using `onClick` on a parent `div`) creates dead zones and confuses screen readers, especially when they contain nested interactive elements like control buttons. The hidden button overlay pattern paired with Tailwind's `:has()` pseudo-class provides an elegant solution, keeping the container fully accessible and maintaining visual interactive states (scale, focus rings) without JavaScript-based hover/active state tracking.
**Action:** When designing clickable cards or complex list items with nested controls, avoid putting `onClick` on the outer `div`. Instead, place a visually hidden, absolutely positioned `<button>` layer behind the controls. Use `pointer-events-none` on presentation wrappers, `pointer-events-auto` on the nested buttons, and style the parent container's active/focus states using `:has([.expand-btn:active])` and `:has([.expand-btn:focus-visible])`.
